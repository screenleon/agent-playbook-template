## Step 4 — Structured preamble

_Assumptions, constraints, and proposed approach stated before any code._

**Assumptions**:
- `last_login` will be set by the auth layer (not this task's scope)
- The column is nullable — existing users have no login history
- The envelope format wraps the response; only the data object changes

**Constraints** (from DECISIONS.md and operating-rules):
- Use raw SQL with sqlc; no ORM — add migration + SQL query
- All API responses follow the envelope format in `docs/api-envelope.md`
- No breaking changes — `last_login` is an additive optional field

**Proposed approach**:
1. Add migration: `ALTER TABLE users ADD COLUMN last_login TIMESTAMPTZ`
2. Update sqlc query to include the new column
3. Update handler to include `last_login` in the response

## Step 9 — Deliverable

_Every task produces this structure. Small tasks may keep it concise; Medium/Large tasks fill it in fully._

```markdown
## Deliverable: Add last_login to user profile API

### Proposal
Added a nullable `last_login TIMESTAMPTZ` column to the `users` table and
exposed it in `GET /users/:id` following the existing sqlc and envelope patterns.

### Alternatives considered
- Compute last_login from an audit log on read (no schema change) — rejected because
  it would require a join on every user read with no indexing benefit.
- Add last_login only to a separate `/users/:id/profile` endpoint — rejected because
  the use case requires it on the primary user object.

### Pros / Cons
| Pros                                     | Cons                                         |
|------------------------------------------|----------------------------------------------|
| Additive, non-breaking                   | Requires migration deploy before code deploy |
| Follows existing sqlc pattern            | Null for existing users until auth sets it   |
| No join, single-column read              |                                              |

### Risks
- Migration must be deployed before the updated handler goes live (deploy order
  risk, medium likelihood, medium impact). Mitigation: deploy migration first in CI
  pipeline.

### Recommendation
Ship as described. Migration first, then handler update.
```

## Step 11 — Mini retrospective

_Required at the end of every task. Feeds into the rolling quality signals._

```text
Friction observed: Deploy order dependency (migration before code) was easy to miss
  when writing the deliverable risks section.
Miss risk: DECISIONS.md contradiction check was most likely to be skipped since
  the change felt obvious.
Most useful rule: Structured preamble forced explicit statement of the
  nullable-column assumption, which would have caused a test failure otherwise.
Next improvement: The deliverable template's Risks section should include a
  "deploy order" prompt for any schema migration task.
```
